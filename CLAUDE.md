# CLAUDE.md

텍사스 로드하우스(TXRH) 주가와 미국 육류 시세를 비교하는 정적 사이트.
배포처는 https://cri8-gemini.github.io/ 이고, 저장소가 곧 사용자 GitHub Pages 저장소다
(`cri8-gemini/cri8-gemini.github.io`).

사용자와의 대화는 한국어로 한다.

## 무엇을 하는 프로젝트인가

원가(육류 시세)가 주가에 반영되는지 눈으로 보려는 것이다. 두 데이터를 주 단위로
맞춰 지수화해 겹쳐 보고, 시차를 줘 가며 상관을 계산한다.

**현재까지의 결론: 뚜렷한 관계가 없다.** 전 구간 주간 변화율 기준으로 네 지표 모두
시차를 줘도 상관이 −0.16 ~ +0.10 범위이고, 시차마다 부호가 뒤집혀 일관된 패턴이
없다. 관측치가 850개대라 잡음 띠가 ±0.07로 좁아 몇몇이 띠를 벗어나지만, 시차 8개를
한꺼번에 보면 하나쯤은 우연히 튀어나온다. 관계로 읽으면 안 된다.

**분기로 묶어서도 봤지만 마찬가지다.** 원가를 분기 평균, 주가를 분기말 종가로 묶어
같은 분기 / 다음 분기 / 2분기 뒤 수익률과 맞춰 봤을 때 상관이 −0.23 ~ +0.16 이고,
분기 66개 기준 잡음 범위(±0.24) 를 벗어나는 조합이 하나도 없다.

주간 시차에서는 전체 구간 +4주가 +0.08 로 부호는 가설(원가가 먼저)과 맞지만, 크기가
너무 작아 실질적 의미를 두기 어렵다. 게다가 부호가 **양수**라는 건 원가가 오르면 주가도
오른다는 뜻이라 원가 전가 경로와는 방향이 반대다. 둘 다 경기 국면에 묶여 움직이는
동조로 보는 편이 자연스럽다.

아직 안 해본 것: 실적 발표일 기준 이벤트 분석, 마진율·동종업체 대비 상대수익률 같은
다른 변수 끌어오기.

## 구조

```
scripts/collect_usmef.py   육류 시세 -> data/usmef_weekly.csv
scripts/collect_txrh.py    TXRH 주봉 -> data/txrh_weekly.csv
scripts/build_site.py      ISO 주차로 병합 -> index.html, data/merged_weekly.csv
scripts/template.html      페이지 원본. 데이터가 빌드 시점에 주입된다
```

전체 재생성:

```bash
python scripts/collect_usmef.py
python scripts/collect_txrh.py
python scripts/build_site.py
```

**`index.html` 을 직접 고치지 말 것.** 생성물이다. `scripts/template.html` 을 고치고
`build_site.py` 를 다시 돌린다. 템플릿의 `/*__DATA__*/null` 자리에 JSON 이 들어간다.

## 데이터 함정 (전부 실제로 당한 것들)

지우면 조용히 틀린 차트가 나온다. 수집기의 방어 코드는 장식이 아니다.

- **범위 밖 연도를 요청하면 오류 대신 최신 연도 데이터가 온다.** `thisYear=2005` 를
  요청하면 2025 년 데이터가 그대로 돌아온다. 처음 수집했을 때 2005~2009 구간이 통째로
  2025 년의 복제본이었고, 차트에 2010 년 수직 절벽으로 나타나서 발견했다. 응답에 실제
  응답 연도가 `thisYear` 로 실려 오므로 `collect_usmef.py` 가 대조해 버린다.
  **확인된 실제 보유 범위는 2010 년부터.**
- **결측이 빈 값이 아니라 `0.00` 으로 온다.** 2017-10-07 의 `priceBeef` 가 그렇고 같은
  행의 나머지 항목은 정상이다. `build_site.py` 의 `to_float` 가 0 이하를 결측 처리한다.
- **Yahoo 의 `adjClose` 는 요청마다 4째 자리가 흔들린다.** 수정계수를 매번 재계산하는
  탓이다. 그대로 저장하면 값이 안 바뀐 날에도 57 행이 바뀐 diff 가 매일 커밋된다.
  소수점 2 자리로 자른다.
- **CSV 는 LF 로 쓴다.** `csv` 모듈 기본 lineterminator 가 CRLF 라, 윈도우 로컬과
  리눅스 러너가 같은 값에 다른 바이트를 만든다. 첫 자동 갱신에서 CSV 두 개가 통째로
  (1,729 줄) 다시 커밋됐다. writer 에 `lineterminator="\n"`, `.gitattributes` 로 이중
  고정.
- 2020 년 5 월 소고기 급등은 **실제 값**이다 (코로나 도축장 셧다운). 주간 25% 를 넘는
  비정상 점프는 전 구간에 없다.

수집기를 손댔으면 **연속 두세 번 돌려서 `git diff -- data` 가 비는지** 확인할 것.
비지 않으면 위 같은 잡음이 새로 들어온 것이다.

## 원본 API

`POST http://usmef.lobonine.gethompy.com/main/mobile_json.php`, body `code=10&thisYear=`

브라우저에서 직접 못 부른다. **평문 http 전용**(https 는 핸드셰이크 실패)이라 https
페이지에서 mixed content 로 막히고, **CORS 헤더도 없다.** 그래서 빌드 시점에 받아
페이지에 박아 넣는 구조다. 이건 우회 불가능하니 "실시간으로 불러오자"는 방향으로
되돌아가지 말 것.

`code` 값은 10 이 시세다. 1·11 은 부위 정보, 3~5 는 앱진, 6~7 은 공지, 13~14 는 관세표.

주가는 Yahoo 차트 API (`query1.finance.yahoo.com/v8/finance/chart/TXRH`, `interval=1wk`).
외부 패키지 없이 표준 라이브러리로 받는다 — CI 에서 의존성 설치가 없다.

## 차트 설계 원칙

바꾸기 전에 이유를 읽을 것. 전부 의도된 제약이다.

- **축을 두 개 쓰지 않는다.** 주가와 시세는 단위도 자릿수도 다르지만, 이중 축은 눈금을
  어떻게 잡느냐로 상관관계를 있어 보이게도 없어 보이게도 만든다. 구간 시작을 100 으로
  맞춘 지수를 한 축에 올린다.
- **세로축 기본은 로그.** 전 구간 기준 TXRH 는 21 배, 육류는 3 배 올랐다. 선형 축에서는
  육류 선이 바닥에 눌려 비교가 안 된다.
- **상관은 수준이 아니라 주간 변화율로.** 둘 다 장기 우상향이라 수준 상관은 아무 관계가
  없어도 1 에 가깝게 나온다.
- **잡음 띠**(관계가 없어도 우연히 나올 수 있는 크기)를 깔고, 그 안쪽 막대는 회색으로
  중립 처리한다. 관측치 수에서 나오는 값이라 시차마다 따로 계산한다.
- **시차는 ±8 주까지.** 기본 구간이 1년(52주)이라 13·26주는 겹치는 구간이 너무 줄어
  상관계수가 흔들린다.
- **시차는 배열 위치가 아니라 실제 ISO 주차로 옮긴다.** 빠진 주가 6곳 있어서(2013 년
  미국 정부 셧다운 3주, 2017 년 2주, 연말 등) "8칸 뒤"와 "8주 뒤"가 어긋난다. 최근
  5년 구간에는 빠진 주가 없어 차이가 없지만, 전체·10년 구간에서는 달라진다.
- **짝짓기를 먼저 하고 구간을 나중에 자른다.** 순서가 반대면 두 선의 지수 기준점이
  서로 다른 주에 놓인다. 이 순서 덕에 1년 구간에서 시차를 줘도 육류 값을 구간 밖에서
  끌어와 관측치가 52주로 유지된다.
- **한 번에 두 계열만.** 색은 파랑(주가)·주황(육류) 고정이고 CVD 검증을 통과한 조합이다.
  계열을 늘리려 했지만 4 색 이상은 색각 이상 분리 기준을 통과하는 조합이 없었다.
  늘려야 하면 색으로 구분하지 말고 facet 으로 쪼갤 것.
- 선 끝 직접 라벨, 막대 값 직접 표기 — 색만으로 계열을 구분하지 않는다.

## 배포

GitHub Actions 가 매일 06:00 KST 에 수집 → **데이터가 바뀐 날만** `data/` 를 커밋 →
사이트를 빌드해 Pages 아티팩트로 배포한다. `workflow_dispatch` 로 수동 실행도 된다.

`index.html` 은 **저장소에 없다** (`.gitignore`). 생성물이라 커밋하면 로컬 작업분과
Actions 가 만든 것이 부딪힌다 (실제로 rebase 충돌이 났다). 로컬에서 `build_site.py` 를
돌려 만들어 보고, 배포는 아티팩트로만 한다.

저장소 설정 두 가지에 의존한다. 배포가 안 되면 여기부터 볼 것:
- Settings → Actions → General → Workflow permissions = **Read and write**
- Settings → Pages → Source = **GitHub Actions**

Source 가 브랜치로 되돌아가면 `pages build and deployment` 라는 내장 워크플로가 같이
돌기 시작한다. 이게 실행 이력에 보이면 설정이 풀린 것이다. 그 상태에서는 저장소에
`index.html` 이 없으므로 사이트가 404 가 된다.

## 이 저장소의 git 설정

사용자는 GitHub 계정이 둘이고, 전역 자격증명(`git:https://github.com`)은 **다른
계정**(`cri8now-alt`)으로 잡혀 있다. 그대로 푸시하면 403 이 난다.

그래서 원격 URL 에 계정을 박아 자격증명을 분리해 뒀다:
`https://cri8-gemini@github.com/cri8-gemini/cri8-gemini.github.io.git`

전역 설정과 다른 계정의 자격증명은 **건드리지 말 것.** 커밋 이름/이메일도 이 저장소
한정으로 `cri8-gemini` + noreply 주소로 잡혀 있다.

## NEWSLINE PDF 수집 (scripts/parse_newsline.py)

USMEF 가 주 1회 usmef.co.kr 에 올리는 NEWSLINE PDF 에서 JSON API 보다 훨씬 넓은
데이터를 뽑는다. `pymupdf` 가 필요하다 (`pip install pymupdf`).

```
data/newsline_cuts.csv    부위별 도매가 (소고기 17종 / 돼지 11종 / 지육 2종)
data/newsline_stats.csv   컷아웃·도축두수·생산량·생우생돈가·수출량
data/newsline_pdf/        원본 캐시. 30MB 라 커밋하지 않는다
```

URL 은 `/ebook/newsline/{연도}/{YYYYMMDD}/{YYYYMMDD}.pdf`.

### 함정

- **없는 날짜에도 HTTP 200 을 준다.** 3KB 짜리 "페이지를 찾을 수 없습니다" PDF 가
  돌아온다. 상태 코드로 판단하면 안 되고 크기로 걸러야 한다. 이걸 모르고 처음엔
  발행일을 수요일로 단정했다가 목요일 발행분(2026-02-19)을 놓쳤다.
- **일부 호는 글자가 벡터로 변환돼 있어 텍스트 추출이 안 된다** (2026 년 기준
  02-25, 06-03, 07-01). 다만 한 호에 5주치가 실리므로 빠진 주는 이웃 호에서
  회수된다 — 33주 연속에 구멍이 없다.
- **표의 월평균 열이 주간 열의 왼쪽에도 오른쪽에도 붙는다.** 주간 열 수도 그 달
  고시 횟수에 따라 4개나 5개다. 그래서 열은 날짜 헤더의 x 좌표로 잡고, 날짜가
  없는 열(=평균)은 자동으로 버린다.
- **라벨과 값의 세로 위치가 표마다 다르다.** 부위 표는 라벨이 값보다 6pt 위,
  지육 표는 14pt 아래다. 행을 좁게 끊은 뒤 '값만 있는 행'과 '라벨만 있는 행'을
  이웃끼리 짝지어 붙인다.
- **증감 방향(▲▼)은 도형이라 추출되지 않는다.** 값과 전주값이 둘 다 있으니
  방향은 계산한다.
- **수출량은 다른 지표보다 열흘 이상 늦게 집계된다.** 가격표 최신 주가 7/18 일 때
  수출 집계는 7/9 까지다. 본문이 밝힌 기간의 끝 날짜를 따로 쓴다.

### 검증 방법 (고칠 때마다 돌릴 것)

- **JSON API 와 대조** — 지육 2종과 116A·123A 는 양쪽에 다 있다. 2026 년 기준
  32주 전량 소수점까지 일치했다.
- **전주값 체인** — 각 호의 '전 주' 값은 직전 호의 본값과 같아야 한다. 인접주
  159건이 일치했고, 어긋난 17건은 전부 생산량·도축두수였다. 이 둘은 PDF 가 스스로
  "USDA 조정수치"라고 밝히는 항목이라 정상이다.

### 알아둘 것

`price123a` 는 필드 순서상 `pricePig` 옆에 있지만 **소고기** 부위다
(123A Short Rib). PDF 대조로 확인했다. 한동안 화면에 "돼지고기 123A" 로 잘못
표시했었다.

### 자동 갱신 (증분)

일일 워크플로가 `parse_newsline.py` 를 인자 없이 돌린다. 전체를 다시 받지 않는다.

- `data/newsline_issues.csv` 에 발행일·크기·파싱 성공 여부를 기록한다. 이 이력이
  있어야 어디부터 이어받을지 알 수 있다.
- 마지막으로 찾은 호 **다음 날부터 오늘까지만** 찔러본다. 없다고 확인된 날짜는
  다시 안 본다(최근 21일은 늦게 올라올 수 있어 예외). 이미지 PDF 였던 호는
  90일 안이면 다시 받아 본다 — 텍스트본으로 재업로드될 수 있다.
- 결과는 기존 CSV 에 병합한다. 나중 호가 이기므로 USDA 수정치가 자동 반영된다.
- 파서를 고쳤으면 `--rebuild --year 2026` 으로 전체 재파싱한다.
- URL 경로의 연도는 `--year` 가 아니라 그 날짜 자신의 연도를 쓴다. 연말을 넘어가면
  탐색 구간이 다음 해로 이어지기 때문.

## 주가 수집 (scripts/collect_stocks.py)

`data/stocks/tickers.txt` 에 적힌 종목을 각각 `data/stocks/<티커>.csv` 로 받는다.
종목을 늘리려면 그 파일에 한 줄 추가하면 된다.

**종목마다 파일을 나눈 이유는 diff 다.** 한 파일에 몰아넣으면 매일 모든 종목의
최신 행이 함께 바뀌어 어느 종목이 갱신됐는지 안 보이고, 한 종목 수집 실패가
나머지까지 휘말린다.

DB 는 쓰지 않는다. 주간 데이터라 종목당 연 52행뿐이고, 20종목을 넣어도 0.5MB 다.
게다가 Pages 는 정적 파일만 서빙하므로 DB 를 두면 서버가 필요해져 지금 구조가
무너지고, git diff 로 "지난주 대비 무엇이 바뀌었나"를 보는 이점도 잃는다.
한계는 저장이 아니라 **페이지 무게**다 — 종목이 5개를 넘어가면 index.html 에
전부 박아 넣는 대신 JSON 을 쪼개 fetch 하는 쪽으로 바꾼다(같은 도메인이라
CORS 문제가 없다).

## 백필

과거 연도는 `--rebuild --year 2025` 처럼 연도별로 돌린다. `--rebuild` 는 "그
연도를 처음부터 다시 훑는다"일 뿐 기존 CSV 를 비우지 않는다. 비우려면 `--reset`
인데, 연도별로 나눠 돌릴 때 쓰면 앞서 모은 해가 날아간다(실제로 한 번 당했다).

발행일 탐색은 요일 우선순위(수→목→화→금→월)로 주당 하나만 찾고 넘어간다.
한 해를 매일 찔러보면 365번인데 이 방식이면 60번 안쪽이다.

확인된 보유 범위는 **2020년부터**(2018·2019 는 없음). 2020~2026 전부 받아 뒀다
— 346주, 가격 10,026건 / 지표 3,832건.

과거 연도를 새로 파싱했으면 **반드시 `python scripts/verify_newsline.py` 를 돌린다.**
세 가지를 서로 다른 근거로 본다: JSON API 대조(가장 강함) / 전주값 체인 / 값 범위.
지금까지 조용히 틀린 것들을 전부 이 방식으로 잡았다.

### 시장동향 카드 파싱 (scripts/newsline_stats.py)

레이아웃이 해마다 바뀌므로 **좌표를 박아 두면 안 된다.** 실제로 겪은 것들:

- **카드 열 위치가 다르다.** 2026년 호는 소·돼지 카드가 모두 오른쪽 열이지만,
  2024년 호는 돼지 카드가 왼쪽 열(폭의 8%)에 있다. 그래서 "카드는 한 섹션에
  4장이 같은 x 로 늘어선다"는 성질로 열을 찾는다.
- **축종을 영문 헤더로 판단하면 안 된다.** 2024년 호에서 실패했다. 같은 열에서
  세로로 가장 가까운 생우/생돈 카드로 정한다.
- **PUA 글리프.** 이 PDF 는 일부 한글 음절을 사용자 정의 영역 문자로 넣는다.
  '100파운드당가격' 이 '100파운드당가격' 으로 추출돼 패턴이 빗나가고,
  그 결과 생우 가격이 통째로 100 으로 읽혔다. 비교 전에 걷어낸다.
- **단위 라벨에 숫자가 붙어 있다** ('100파운드당가격'). 레이아웃에 따라 값 자리
  안으로 들어오므로 걸러야 한다.
- **수출량 문구가 해마다 다르다.** 숫자 앞에 서술이 끼고("전 주 대비 14% 감소한
  28,920톤"), 띄어쓰기도 흔들린다. 숫자를 바로 집지 말고 '톤' 앞의 숫자를 찾는다.
- **활자 합자와 기호 표기가 호마다 다르다.** 옛 호는 'neck-oﬀ'(ﬀ 합자)와
  '1x1'(ASCII x), 최근 호는 'neck-off'와 '1×1'(곱셈기호)을 쓴다. 정규화하지 않으면
  같은 부위가 서로 다른 항목으로 갈린다(실제로 116A 가 반으로 쪼개져 있었다).
  NFKC 정규화 후 × -> x 로 바꾼다.
- **카드 간격이 호마다 다르다.** 2026년은 240pt, 2020년은 121pt 다. 전주값을 찾는
  구간을 고정하면 촘촘한 호에서 다음 카드 값을 끌어온다. 같은 열의 다음 카드
  직전까지로 잘라야 한다.
- **전주값 줄의 문구도 다르다** — '전 주', '전주', '지난주'.
- **2025년 10~11월 생산량은 실제로 없다.** 미국 정부 셧다운으로 USDA 가 발표를
  못 했고, PDF 에 `-억 ----만` 으로 찍혀 있다. 파서가 결측으로 처리한다.

검증은 전주값 체인으로 한다. 현재 일치율 90.3%(863/93)이고, 남은 불일치는 대부분
생산량·도축두수 — PDF 가 스스로 "USDA 조정수치"라고 밝히는 항목이다.

## 원본 오타 (build_site.py 에서 거른다)

PDF 에 자릿수가 빠진 오타가 있다. 2022-12-24 립아이가 `2.45` 로 찍혀 있는데 같은
행의 앞뒤 주는 `24.47 / 27.32` 다. **같은 주를 싣는 네 개 호가 모두 같은 오타라
이웃 호로도 회수되지 않는다.**

한 점만 2배 이상 튀고 그 앞뒤가 서로 15% 안쪽으로 붙어 있을 때만 버린다. 코로나
폭락처럼 여러 주에 걸친 움직임이나 50% 수준의 변동은 걸리지 않는다. 조건을 넓히면
진짜 급등락까지 지운다.

`newsline_cuts.csv` 는 원본 그대로 둔다 — 거르는 건 차트를 만들 때뿐이다.

## 분석 결과 (2026-07 기준, 표본 345주)

**원가 -> TXRH 주가 관계는 확인되지 않는다.** 다만 "도구가 부족해서"가 아니다.

- **대조군은 통과한다.** 공급 -> 컷아웃 가격이 1주 뒤에 네 조합 모두 음수이고
  잡음 띠 밖이다(소 생산량 -0.14, 소 도축 -0.13, 돼지 생산량 -0.12, 돼지 도축
  -0.13). 표본 100개대에서는 못 잡았고 300개에서 잡힌다. **한때 "주간 변화율이라는
  도구가 안 맞는다"고 판단했는데 틀렸다 — 표본 문제였다.**
- **스테이크 부위도 특별하지 않다.** 립아이 -0.08, 스트립로인 -0.01, 텐더로인
  -0.03, 탑버트 -0.03 (동시점, 시장 성분 제거 후). 전부 띠 안이다.
- **큰 값들은 원가 효과가 아니다.** 시차별로 부호가 규칙적으로 교대하는데
  (-8주 음수 / -4주 양수 / +4주 양수 / +8주 음수), 이는 육류 계열 자체의 성질이다
  — 가격 변화율이 1주 뒤와 +0.59, 4주 뒤와 -0.30 으로 8주 주기로 진동한다.
  결정적으로 **같은 패턴이 SPY(시장 전체)에서 더 강하게 나온다**(브리스킷 TXRH
  -0.30 vs SPY -0.33). SPY 는 TXRH 의 소고기 원가와 무관하므로 원가 전가일 수 없다.

### 시차 8주 이상은 보지 않는다

`+8주` 에서는 12종 × 컷아웃 2개 = **24개 조합이 하나도 빠짐없이 음수**로 나온다
(평균 -0.136). 소고기와 무관한 SPY(-0.10)와 도미노(-0.04)까지 같은 방향이다.
특정 종목이 원가에 민감해서가 아니라 육류 계열의 8주 주기 진동이 만드는 값이라,
원가 효과("원가 오르면 주가 내림")로 오독하기 딱 좋다.

시차를 0~4주로 한 주 단위로 좁히니 띠 밖 조합이 크게 줄었다 — 2023년 이후는
27개/1200회로 **우연 기대치(60개)보다도 적고**, 전 구간도 123개/1200회다.
남은 음수 상위도 CAVA(표본 153주, 띠 못 넘음), 도미노 vs 플랭크스테이크,
DRI vs 소 한국수출처럼 경로가 없는 조합들이다.

인접 시차끼리 부호가 이어지는지로 진짜 신호를 가릴 수 있으니, 이 범위를 유지한다.

그래서 화면에서 음수 시차를 뺐다. 원가가 주가를 끌고 간다는 가설과 방향이 반대고,
거기서 크게 나오던 값들이 원가와 무관한 것으로 확인됐다.

이상한 결과가 아니다. 주가는 매출·점포수·마진·시장 심리를 한꺼번에 반영하고
원가는 그중 일부다. 주간 단위로 그 조각만 떼어 보이려면 훨씬 큰 효과여야 한다.

### 동종업체 비교 (2026-07)

외식주 14종을 모아 육류 계열과의 연관을 쟀다. 방법이 결과를 좌우한 사례라 남겨 둔다.

- **한 계열로만 재면 착시가 생긴다.** 소 지육 하나로 재니 스테이크하우스 4종이
  모두 +4주에서 띠를 넘어 "노출도 순으로 갈린다"고 보였다. 컷아웃(같은 시장의
  다른 측정치)으로 다시 재니 7종이 전부 띠 안으로 들어갔다. 두 계열 모두에서
  살아남은 건 14종 중 1개뿐 — 14종을 검정했으니 우연으로도 나올 수다.
- **전 계열(41개) × 시차 3개로 넓히면 판별이 된다.** 걸린 횟수가 25회(CAKE)부터
  1회(CMG)까지 벌어진다. 우연 기대치는 6.2회다.
- **위약 검정으로 확인했다.** 육류 계열의 시간 순서를 섞으면 어느 종목이든
  6~7회로 수렴한다. 실제 데이터의 25/21/18 은 종목 자체의 성질이 아니라 육류
  계열과의 시간 정렬에서 나온다.

### 기간을 갈라 보니 코로나 현상이었다

2020~2022 와 2023~ 로 나누면(각 156주 / 168주) 갈린다. 크래커배럴 5.2배 -> 0.8배,
TXRH 3.6 -> 1.0, 치즈케이크 3.4 -> 1.1, STK 3.3 -> 0.2. **후반부는 일곱 종목 전부
1.0배 근처**다. 표본 부족이 아니다 — 후반이 오히려 168주로 더 길다.

2020~2022 는 도축장 셧다운으로 육류값이 폭등하고 외식주가 폐쇄·재개장으로
요동친 시기다. **두 계열이 같은 충격에 함께 반응한 것**이지 원가가 주가를 끈 게
아니다.

### 그런데 2023년 이후 구간은 판정 자체가 안 된다

대조군인 "공급 -> 가격" 관계가 이 구간에서 사라진다. 전 구간 1주 뒤 -0.13 ~ -0.14
(네 조합 모두)가 2023년 이후에는 +0.01 ~ +0.03 이다. **반드시 존재해야 할 관계도
검출되지 않으므로**, 같은 구간의 "주가와 관계 없음" 도 강하게 주장할 수 없다.
164주에 잡음 띠가 ±0.15 라 웬만한 크기는 못 잡는다.

실제로 2023년 이후로만 15종을 다시 재면 **도미노가 1위(3.3배)** 로 올라온다.
피자 체인이라 소고기 노출이 없고 전 구간에서는 0.8배로 꼴찌권이던 종목이다.
위약 검정에서도 실제와 섞은 데이터가 구별되지 않는다(CAKE 실제 7 vs 섞음 3·5·7).

**그래서 종목은 통계가 아니라 업태로 골랐다.** 통계로 고르면 잡음의 순위를
고르는 셈이다. 목록은 `data/stocks/tickers.txt` 와 `build_site.py` 의 `STOCKS` 에
있고, 대조군(도미노·웬디스·카바·SPY)을 함께 두어 화면에서 눈으로 갈라 볼 수 있게
했다.

판정하려면 2023년 이후가 300주쯤 쌓여야 하고, 지금 속도로 2029년경이다.

이 기준으로 고른 종목이 `data/stocks/tickers.txt` 에 있다. **"TXRH 와 주가가
비슷한 순"과는 다른 목록이다** — 다든(DRI)은 TXRH 와 상관 0.67 로 가장 비슷하지만
육류 연관은 1.5배로 낮아 뺐다.

부호는 여전히 **양수**다(육류값이 오르면 4주 뒤 주가도 오른다). 원가 전가라면
음수여야 하므로, 육류 시세가 외식 수요의 대리지표로 움직이는 쪽으로 읽는 게
자연스럽다. 게다가 갈리는 축이 소고기 노출도가 아니라 **착석형 캐주얼다이닝 vs
패스트푸드**에 가깝다 — 웬디스·맥도날드는 소고기 노출이 큰데도 3회·7회로 낮다.

다음에 시도한다면 실적 발표일 기준 이벤트 분석이나 마진율 같은 다른 변수다.

## NEWSLINE PDF 수집 (scripts/parse_newsline.py)

USMEF 가 주 1회 usmef.co.kr 에 올리는 NEWSLINE PDF 에서 JSON API 보다 훨씬 넓은
데이터를 뽑는다. `pymupdf` 가 필요하다 (`pip install pymupdf`).

```
data/newsline_cuts.csv    부위별 도매가 (소고기 17종 / 돼지 11종 / 지육 2종)
data/newsline_stats.csv   컷아웃·도축두수·생산량·생우생돈가·수출량
data/newsline_pdf/        원본 캐시. 30MB 라 커밋하지 않는다
```

URL 은 `/ebook/newsline/{연도}/{YYYYMMDD}/{YYYYMMDD}.pdf`.

### 함정

- **없는 날짜에도 HTTP 200 을 준다.** 3KB 짜리 "페이지를 찾을 수 없습니다" PDF 가
  돌아온다. 상태 코드로 판단하면 안 되고 크기로 걸러야 한다. 이걸 모르고 처음엔
  발행일을 수요일로 단정했다가 목요일 발행분(2026-02-19)을 놓쳤다.
- **일부 호는 글자가 벡터로 변환돼 있어 텍스트 추출이 안 된다** (2026 년 기준
  02-25, 06-03, 07-01). 다만 한 호에 5주치가 실리므로 빠진 주는 이웃 호에서
  회수된다 — 33주 연속에 구멍이 없다.
- **표의 월평균 열이 주간 열의 왼쪽에도 오른쪽에도 붙는다.** 주간 열 수도 그 달
  고시 횟수에 따라 4개나 5개다. 그래서 열은 날짜 헤더의 x 좌표로 잡고, 날짜가
  없는 열(=평균)은 자동으로 버린다.
- **라벨과 값의 세로 위치가 표마다 다르다.** 부위 표는 라벨이 값보다 6pt 위,
  지육 표는 14pt 아래다. 행을 좁게 끊은 뒤 '값만 있는 행'과 '라벨만 있는 행'을
  이웃끼리 짝지어 붙인다.
- **증감 방향(▲▼)은 도형이라 추출되지 않는다.** 값과 전주값이 둘 다 있으니
  방향은 계산한다.
- **수출량은 다른 지표보다 열흘 이상 늦게 집계된다.** 가격표 최신 주가 7/18 일 때
  수출 집계는 7/9 까지다. 본문이 밝힌 기간의 끝 날짜를 따로 쓴다.

### 검증 방법 (고칠 때마다 돌릴 것)

- **JSON API 와 대조** — 지육 2종과 116A·123A 는 양쪽에 다 있다. 2026 년 기준
  32주 전량 소수점까지 일치했다.
- **전주값 체인** — 각 호의 '전 주' 값은 직전 호의 본값과 같아야 한다. 인접주
  159건이 일치했고, 어긋난 17건은 전부 생산량·도축두수였다. 이 둘은 PDF 가 스스로
  "USDA 조정수치"라고 밝히는 항목이라 정상이다.

### 알아둘 것

`price123a` 는 필드 순서상 `pricePig` 옆에 있지만 **소고기** 부위다
(123A Short Rib). PDF 대조로 확인했다. 한동안 화면에 "돼지고기 123A" 로 잘못
표시했었다.

### 자동 갱신 (증분)

일일 워크플로가 `parse_newsline.py` 를 인자 없이 돌린다. 전체를 다시 받지 않는다.

- `data/newsline_issues.csv` 에 발행일·크기·파싱 성공 여부를 기록한다. 이 이력이
  있어야 어디부터 이어받을지 알 수 있다.
- 마지막으로 찾은 호 **다음 날부터 오늘까지만** 찔러본다. 없다고 확인된 날짜는
  다시 안 본다(최근 21일은 늦게 올라올 수 있어 예외). 이미지 PDF 였던 호는
  90일 안이면 다시 받아 본다 — 텍스트본으로 재업로드될 수 있다.
- 결과는 기존 CSV 에 병합한다. 나중 호가 이기므로 USDA 수정치가 자동 반영된다.
- 파서를 고쳤으면 `--rebuild --year 2026` 으로 전체 재파싱한다.
- URL 경로의 연도는 `--year` 가 아니라 그 날짜 자신의 연도를 쓴다. 연말을 넘어가면
  탐색 구간이 다음 해로 이어지기 때문.

## 주가 수집 (scripts/collect_stocks.py)

`data/stocks/tickers.txt` 에 적힌 종목을 각각 `data/stocks/<티커>.csv` 로 받는다.
종목을 늘리려면 그 파일에 한 줄 추가하면 된다.

**종목마다 파일을 나눈 이유는 diff 다.** 한 파일에 몰아넣으면 매일 모든 종목의
최신 행이 함께 바뀌어 어느 종목이 갱신됐는지 안 보이고, 한 종목 수집 실패가
나머지까지 휘말린다.

DB 는 쓰지 않는다. 주간 데이터라 종목당 연 52행뿐이고, 20종목을 넣어도 0.5MB 다.
게다가 Pages 는 정적 파일만 서빙하므로 DB 를 두면 서버가 필요해져 지금 구조가
무너지고, git diff 로 "지난주 대비 무엇이 바뀌었나"를 보는 이점도 잃는다.
한계는 저장이 아니라 **페이지 무게**다 — 종목이 5개를 넘어가면 index.html 에
전부 박아 넣는 대신 JSON 을 쪼개 fetch 하는 쪽으로 바꾼다(같은 도메인이라
CORS 문제가 없다).

## 백필

과거 연도는 `--rebuild --year 2025` 처럼 연도별로 돌린다. `--rebuild` 는 "그
연도를 처음부터 다시 훑는다"일 뿐 기존 CSV 를 비우지 않는다. 비우려면 `--reset`
인데, 연도별로 나눠 돌릴 때 쓰면 앞서 모은 해가 날아간다(실제로 한 번 당했다).

발행일 탐색은 요일 우선순위(수→목→화→금→월)로 주당 하나만 찾고 넘어간다.
한 해를 매일 찔러보면 365번인데 이 방식이면 60번 안쪽이다.

확인된 보유 범위는 **2020년부터**(2018·2019 는 없음). 2020~2026 전부 받아 뒀다
— 346주, 가격 10,026건 / 지표 3,832건.

과거 연도를 새로 파싱했으면 **반드시 `python scripts/verify_newsline.py` 를 돌린다.**
세 가지를 서로 다른 근거로 본다: JSON API 대조(가장 강함) / 전주값 체인 / 값 범위.
지금까지 조용히 틀린 것들을 전부 이 방식으로 잡았다.

### 시장동향 카드 파싱 (scripts/newsline_stats.py)

레이아웃이 해마다 바뀌므로 **좌표를 박아 두면 안 된다.** 실제로 겪은 것들:

- **카드 열 위치가 다르다.** 2026년 호는 소·돼지 카드가 모두 오른쪽 열이지만,
  2024년 호는 돼지 카드가 왼쪽 열(폭의 8%)에 있다. 그래서 "카드는 한 섹션에
  4장이 같은 x 로 늘어선다"는 성질로 열을 찾는다.
- **축종을 영문 헤더로 판단하면 안 된다.** 2024년 호에서 실패했다. 같은 열에서
  세로로 가장 가까운 생우/생돈 카드로 정한다.
- **PUA 글리프.** 이 PDF 는 일부 한글 음절을 사용자 정의 영역 문자로 넣는다.
  '100파운드당가격' 이 '100파운드당가격' 으로 추출돼 패턴이 빗나가고,
  그 결과 생우 가격이 통째로 100 으로 읽혔다. 비교 전에 걷어낸다.
- **단위 라벨에 숫자가 붙어 있다** ('100파운드당가격'). 레이아웃에 따라 값 자리
  안으로 들어오므로 걸러야 한다.
- **수출량 문구가 해마다 다르다.** 숫자 앞에 서술이 끼고("전 주 대비 14% 감소한
  28,920톤"), 띄어쓰기도 흔들린다. 숫자를 바로 집지 말고 '톤' 앞의 숫자를 찾는다.
- **활자 합자와 기호 표기가 호마다 다르다.** 옛 호는 'neck-oﬀ'(ﬀ 합자)와
  '1x1'(ASCII x), 최근 호는 'neck-off'와 '1×1'(곱셈기호)을 쓴다. 정규화하지 않으면
  같은 부위가 서로 다른 항목으로 갈린다(실제로 116A 가 반으로 쪼개져 있었다).
  NFKC 정규화 후 × -> x 로 바꾼다.
- **카드 간격이 호마다 다르다.** 2026년은 240pt, 2020년은 121pt 다. 전주값을 찾는
  구간을 고정하면 촘촘한 호에서 다음 카드 값을 끌어온다. 같은 열의 다음 카드
  직전까지로 잘라야 한다.
- **전주값 줄의 문구도 다르다** — '전 주', '전주', '지난주'.
- **2025년 10~11월 생산량은 실제로 없다.** 미국 정부 셧다운으로 USDA 가 발표를
  못 했고, PDF 에 `-억 ----만` 으로 찍혀 있다. 파서가 결측으로 처리한다.

검증은 전주값 체인으로 한다. 현재 일치율 90.3%(863/93)이고, 남은 불일치는 대부분
생산량·도축두수 — PDF 가 스스로 "USDA 조정수치"라고 밝히는 항목이다.

## 원본 오타 (build_site.py 에서 거른다)

PDF 에 자릿수가 빠진 오타가 있다. 2022-12-24 립아이가 `2.45` 로 찍혀 있는데 같은
행의 앞뒤 주는 `24.47 / 27.32` 다. **같은 주를 싣는 네 개 호가 모두 같은 오타라
이웃 호로도 회수되지 않는다.**

한 점만 2배 이상 튀고 그 앞뒤가 서로 15% 안쪽으로 붙어 있을 때만 버린다. 코로나
폭락처럼 여러 주에 걸친 움직임이나 50% 수준의 변동은 걸리지 않는다. 조건을 넓히면
진짜 급등락까지 지운다.

`newsline_cuts.csv` 는 원본 그대로 둔다 — 거르는 건 차트를 만들 때뿐이다.

## 분석 결과 (2026-07 기준)

원가와 TXRH 주가 사이에서 **주간 변화율로는 어떤 관계도 잡히지 않는다.** 네 번
시도했고 모두 같은 결론이다: 주간 시차(±8주), 분기 묶기, 부위별 세분화(17종),
공급 지표(도축두수·생산량·컷아웃·수출량).

**중요한 단서**: 주간 변화율은 "공급이 늘면 가격이 내린다"는 교과서적 관계조차
잡아내지 못한다(소 생산량 -> 컷아웃 r=-0.10, 띠 ±0.19). 관계가 없다기보다
**주 단위 변화율이라는 도구 자체가 이 데이터에 안 맞는다**고 보는 편이 맞다.
주간 값이 USDA 집계 시점·조정에 흔들리는 폭이 실제 신호보다 크다.

다음에 시도한다면 변화율이 아니라 **수준(level)의 추세**나 더 긴 구간 평균,
혹은 실적 발표일 기준 이벤트 분석 쪽이다.

SPY 는 시장 기준선이라 목록에 넣어 뒀다. 어떤 값이 종목 고유인지 시장 전체의
움직임인지 화면에서 바로 갈라 볼 수 있다 — 실제로 "원가 효과처럼 보이던 값이
SPY 에서 더 크게 나온다"는 것을 이 비교로 확인했다.

## 시장동향 지표를 비교 차트에 넣기

컷아웃·생축가·도축두수·생산량·수출량을 별도 카드로 따로 그렸다가, 위 비교
차트의 지표 선택지로 합쳤다. 그래야 시차·로그축·표까지 같은 도구를 쓸 수 있다.

`newsline_stats.csv` 를 ISO 주차 키로 `merged_weekly.csv` 에 컬럼으로 합친다
(`beef_cutout`, `pork_slaughter` …). 지표가 14개라 버튼은 **가격 / 수급** 두 줄로
나누되 고르는 값은 하나다.

지표마다 구간이 다르다 — 가격 4계열은 2010년부터, PDF 에서 온 지표는 2023-12
부터다. 짝짓기에서 자동으로 처리되므로 따로 손댈 것은 없다.

### 화면을 만들며 밟은 함정

- **함수 이름 충돌.** 요약 타일용 `renderStats(points)` 가 이미 있는데 같은 이름을
  새로 정의해 뒤 카드까지 통째로 멈췄다. 예외는 나지 않고 조용히 안 그려진다.
- **격자 폭 측정 시점.** 차트를 하나씩 붙이며 `clientWidth` 를 재면 첫 차트만
  혼자 있는 상태의 폭으로 잡혀 축척이 어긋난다. 다 붙인 뒤 재야 한다.
- **연도 라벨 겹침.** 구간 맨 앞에 몇 주뿐인 해가 있으면 다음 해와 겹쳐 "20234"
  로 찍힌다. 픽셀 간격으로 한 번 더 거른다.
- 도축두수·생산량의 톱니는 오류가 아니라 연휴 주 감소다(추수감사절·성탄절·독립기념일).
